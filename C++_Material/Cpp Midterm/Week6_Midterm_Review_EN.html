C. Must-Know Code Templates

1. Proper Initialization List (avoid self-assignment bug)

class Student {
    int id;
    string name;
public:
    // WRONG: self-assignment bug!
    Student(int id, string name) {
        id = id;        // Parameter assigned to itself!
        name = name;    // Member stays uninitialized!
    }
    
    // CORRECT: Use initialization list
    Student(int id, string name) 
        : id(id), name(name) {}
    
    // ALSO CORRECT: Use this->
    Student(int id, string name) {
        this->id = id;
        this->name = name;
    }
};

2. Constructor with Validation

class Course {
    string code;
    int capacity;
public:
    Course(string c, int cap)
        : code(c.empty() ? "UNKNOWN" : c)  // Inline validation
        , capacity(cap > 0 ? cap : 30)      // Ensure positive
    {
        if (c.empty()) {
            cout << "Warning: Empty code\n";
        }
    }
};

3. Copy Constructor

class Professor {
    int id;
    string name;
    string courses[50];
    int count;
public:
    Professor(const Professor& other)
        : id(other.id)
        , name(other.name)
        , count(other.count)
    {
        // Deep copy for arrays
        for (int i = 0; i < count; i++) {
            courses[i] = other.courses[i];
        }
    }
};

4. Static Member for Tracking

class Tracker {
    static int totalCount;  // Declaration only!
    string name;
public:
    Tracker(string n) : name(n) {
        totalCount++;
    }
    ~Tracker() {
        totalCount--;
    }
    static int getTotal() { return totalCount; }
};

// REQUIRED: Definition outside class
int Tracker::totalCount = 0;
Common trap: Forgetting int ClassName::member = value; causes linker error!

D. Hard Parts & Common Mistakes

Hard Point 1: Self-assignment shadowing. When parameter name equals member name, id = id; assigns parameter to itself, leaving member uninitialized.
Fix: Use initialization list : id(id) or this->id = id;
Hard Point 2: Confusing initialization order with initialization list order. Real rule: members initialize in declaration order, NOT the order in init list!
Hard Point 3: Array default constructor requirement. When creating Class arr[10];, compiler calls default constructor 10 times automatically. No default ctor = compilation error.
Hard Point 4: Static local variables persist between function calls. They are created only once (first call) and keep their value.
Quick anti-mistake checklist: Check for self-assignment → Check init list usage → Verify default ctor for arrays → Trace constructor/destructor order.

E. Practice Zone

Q1. Spot the Bug: Self-Assignment Basic

What's wrong with this constructor? How would you fix it?

class Student {
    int id;
    string name;
public:
    Student(int id, string name) {
        id = id;
        name = name;
    }
};

Answer: Parameter shadowing causes self-assignment. The parameter id is assigned to itself, leaving the member id uninitialized!

// FIX 1: Use initialization list
Student(int id, string name) 
    : id(id), name(name) {}

// FIX 2: Use this->
Student(int id, string name) {
    this->id = id;
    this->name = name;
}
Memory Tip: "Shadow trap" — when parameter has same name as member, the local variable (parameter) hides the member. Always use init list or this-> to be safe!

Q2. Constructor/Destructor Order Core

What is the exact output of this program?

class Trace {
    string label;
public:
    Trace(string s) : label(s) {
        cout << "Create: " << label << endl;
    }
    ~Trace() {
        cout << "Destroy: " << label << endl;
    }
};

void func() {
    Trace a("A");
    {
        Trace b("B");
        Trace c("C");
    }
    Trace d("D");
}

int main() {
    cout << "Start" << endl;
    func();
    cout << "End" << endl;
    return 0;
}
Start
Create: A
Create: B
Create: C
Destroy: C
Destroy: B
Create: D
Destroy: D
Destroy: A
End

Explanation:

  • Objects created in order: A (outer scope), then B, C (inner scope), then D (outer scope again)
  • When leaving inner scope {}, C and B are destroyed in reverse order (LIFO)
  • When leaving func(), D and A are destroyed in reverse order
Memory Tip: "Last In, First Out" — like a stack of plates. The last object created in a scope is the first one destroyed when leaving that scope.

Q5. Array Default Constructor Core

Why does this fail to compile? Fix it.

class Student {
    int id;
    string name;
public:
    Student(int i, string n) : id(i), name(n) {}
    void print() const;
};

int main() {
    Student roster[30];  // ERROR!
    return 0;
}

Problem: Arrays require a default constructor because Student roster[30] needs to construct 30 objects without parameters.

// FIX: Add default constructor
class Student {
    int id;
    string name;
public:
    Student() : id(0), name("Unknown") {}  // REQUIRED!
    Student(int i, string n) : id(i), name(n) {}
    void print() const;
};

int main() {
    Student roster[30];           // OK: Calls default ctor 30 times
    roster[0] = Student(1, "A");  // Can assign specific values later
    return 0;  // 30 destructors called in reverse order
}
Rule to Remember: "Array = Default Constructor Required" — When creating arrays of objects, the default constructor is automatically called for each element.

Q6. Initialization Order Challenge

What does this print? Explain why.

class OrderDemo {
    int a;  // Declared FIRST
    int b;  // Declared SECOND
    int c;  // Declared THIRD
public:
    OrderDemo() 
        : c(30)  // Written first...
        , a(10)  // Written second...
        , b(20)  // Written third...
    {
        cout << a << " " << b << " " << c << endl;
    }
};

Answer: 10 20 30

Explanation: Members are initialized in declaration order (a→b→c), NOT the order in the initialization list!

  • Even though init list writes c, a, b...
  • Actual initialization: a(10) first (declared first)
  • Then b(20) (declared second)
  • Then c(30) (declared third)
Best Practice: Always write initialization list in the same order as member declarations to avoid confusion!
