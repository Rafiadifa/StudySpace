Building the Shape Hierarchy
C++ Final10 June 2026
The four moving parts. (1) An abstract base = a class with at least one pure virtual = 0 method — it can't be instantiated. (2) Override in each derived class. (3) A virtual destructor so delete base_ptr destroys the full object. (4) Dynamic dispatch: calling through a Shape* runs the derived area() via the vtable.
Problem 1 The abstract base Step 1
Write Shape with a protected name, a pure virtual area(), and a virtual destructor. Why can't you write Shape s("x");?
Answer 
class Shape {
protected:
    string name;
public:
    Shape(const string& n) : name(n) {}
    virtual double area() const = 0;  // pure virtual ⇒ abstract
    virtual ~Shape() {}                 // virtual dtor
};
The = 0 makes Shape abstract — it has no complete implementation, so it can't be instantiated. It exists only to be inherited and to provide a common interface pointer type.
Problem 2 Circle & Rectangle override Step 2
Derive Circle (radius) and Rectangle (w, h). Override area() in each. Use the override keyword and explain what it protects against.
Answer 
class Circle : public Shape {
    double r;
public:
    Circle(const string& n, double r) : Shape(n), r(r) {}
    double area() const override { return 3.14159265358979 * r * r; }
};
class Rectangle : public Shape {
    double w, h;
public:
    Rectangle(const string& n, double w, double h) : Shape(n), w(w), h(h) {}
    double area() const override { return w * h; }
};
override tells the compiler "this must be replacing a virtual base method." If you mistype the signature (e.g. forget const), you'd silently create a new method instead of overriding — override turns that into a compile error.
Problem 3 Dynamic dispatch Step 3
Store mixed shapes in vector<Shape*> and print each area. What does s->area() actually call, and what makes that work?
Answer 
vector<Shape*> shapes;
shapes.push_back(new Circle("C1", 2));
shapes.push_back(new Rectangle("R1", 3, 4));
for (Shape* s : shapes)
    cout << s->name() << " " << s->area() << "\n";
for (Shape* s : shapes) delete s;   // virtual dtor cleans up fully
Even though the static type is Shape*, area() is virtual, so the call dispatches at runtime to the real object's override (Circle's or Rectangle's). This is polymorphism — one loop, many area formulas. Without the virtual destructor, delete s would only run ~Shape and leak the derived part.
Problem 4 Object slicing trap Trap
What goes wrong with vector<Shape> (by value) instead of vector<Shape*>? Why does dispatch break?
Answer You can't even hold Shape by value (it's abstract) — but the deeper lesson is object slicing: copying a Circle into a Shape variable keeps only the base part and discards r. Polymorphism needs a pointer or reference to the base, never a by-value base object.
Rule: store polymorphic objects via Shape* (or better, unique_ptr<Shape>). By-value containers slice and freeze the type.
