Part 2: Inheritance in C++

Exercise 2.1 Intermediate

Base Subobject Initialization

Person has Person(int id, const string& name). Student adds gpa. Which initializer list is correct for Student(int id, const string& name, double gpa)?

class Student : public Person { double gpa; public: Student(int id, const string& name, double gpa); };
Hint: The base class subobject must be constructed via the member-initializer list, not by calling the base constructor like a normal function in the body. Order in the list does not override the rule that bases are initialized before members.
Solution

Answer: C — Use Person(id, name) in the initializer list. You cannot "call" Person(...) as a statement in the constructor body to construct the base. Option D’s id = id is also wrong for base members in typical designs.

Exercise 2.2 Intermediate

Constructor / Destructor Order

Given Student publicly derived from Person, each constructor and destructor prints one line. What is the complete print order when Student s; is created and then goes out of scope at the end of main?

// Person ctor prints: 1. Person constructor // Student ctor prints: 2. Student constructor // Student dtor prints: 3. Student destructor // Person dtor prints: 4. Person destructor
Hint: Construction is foundation-first (base → derived). Destruction is the opposite: remove the roof before the foundation (derived → base).
Solution

Answer: A — Construct base, then derived. Destroy derived, then base. Matches the "house" analogy in the lecture.

Exercise 2.3 Intermediate

Trace construction and destruction for C obj; (from the lecture's three-level example). What is the complete output order?

class A { public: A() { cout << "A constructed" << endl; } ~A() { cout << "A destroyed" << endl; } }; class B : public A { public: B() { cout << "B constructed" << endl; } ~B() { cout << "B destroyed" << endl; } }; class C : public B { public: C() { cout << "C constructed" << endl; } ~C() { cout << "C destroyed" << endl; } }; int main() { C obj; return 0; }
Hint: Every base subobject is constructed before the derived class body runs; destruction unwinds in the reverse order.
Solution

Answer: A — Output: A constructed, B constructed, C constructed, then C destroyed, B destroyed, A destroyed.

Exercise 2.5 Intermediate

Code Fix: const Member Initialization

The lecture stresses that const data members must be initialized in the constructor initializer list, not assigned in the body. List at least two distinct problems with this code.

class ChessPiece { private: const string pieceID; const bool isWhite; public: ChessPiece(string id, bool white) { pieceID = id; // line 1 isWhite = white; // line 2 } };
Hint: After the opening {, all subobjects are already supposed to be constructed; const members cannot be "assigned" later like ordinary fields.
Solution
  1. const members must be initialized in the initializer list, e.g. ChessPiece(string id, bool white) : pieceID(id), isWhite(white) { }.
  2. Assignments like pieceID = id; in the constructor body are compile errors because const cannot be modified after initialization.
ChessPiece(string id, bool white) : pieceID(id), isWhite(white) { }

Part 3: Static Members & Common Mistakes

Exercise 3.1 Advanced

Static Member Pitfalls

The review section warns about static data members. List at least three distinct problems with the code below (one short phrase per line is enough).

class ChessClub { private: static int memberCount; public: ChessClub() { static int memberCount = 0; // line A memberCount++; } }; int static ChessClub::memberCount = 0; // line B
Hint: A static member is one shared variable for the whole class — it should not be re-declared as a local static inside each constructor. Also check how the out-of-class definition must look, and whether resetting the counter in the constructor makes sense.
Solution
  1. Line A: Declaring static int memberCount = 0 inside the constructor creates a different local static variable, shadowing the class member — wrong pattern for counting all objects.
  2. Line A: If interpreted as resetting shared state each construction, that would defeat a cumulative count (the lecture warns against initializing shared statics in the ctor body).
  3. Line B: The out-of-class definition must be int ChessClub::memberCount = 0; — do not repeat the static keyword there (compile error).

Key insight: Define the static data member exactly once outside the class (in one .cpp), without static on the definition line.
