MVC + Exception Handling
C++ Final10 June 2026
MVC in one line. The Model holds data + rules, the View shows it, the Controller turns user input into Model updates. The point: the Model never knows how it's displayed, so you can swap the View without touching the logic.

Model

State + business rules. No I/O, no cout. Pure data and validation.

View

Renders the Model. Reads state, prints it. No logic.

Controller

Takes input, calls Model methods, picks the View. The glue.

Problem 1 Which layer owns it? Concept
For a to-do app, assign each to M / V / C: (a) "the list of tasks", (b) "printing the menu", (c) "validating that a due-date isn't in the past", (d) "reading the user's keypress".
Answer (a) Model — it's the data. (b) View — pure presentation. (c) Model — a business rule lives with the data it guards. (d) Controller — input handling routes to Model actions.
Rule of thumb: validation belongs to the Model, not the Controller — that way the rule holds no matter which Controller or View calls it.
Problem 2 throw / try / catch Mechanics
Write a withdraw that throws std::runtime_error on overdraft, and a caller that catches it and prints the message. Trace what happens to the stack when the throw fires.
Answer 
#include <stdexcept>
void withdraw(double& bal, double amt) {
    if (amt > bal)
        throw runtime_error("Insufficient funds");
    bal -= amt;
}

int main() {
    double bal = 100;
    try {
        withdraw(bal, 250);
    } catch (const exception& e) {
        cout << "Error: " << e.what() << "\n";
    }
}
Stack unwinding: when throw fires, control leaves withdraw immediately, destructors run for locals along the way, and execution resumes at the matching catch. Catch by const& to avoid slicing the exception object.
Problem 3 Catch order & specificity Trap
A function may throw both std::out_of_range and a generic std::exception. In what order must the catch blocks appear, and why?
Answer Most-derived first, base last. catch (const out_of_range&) before catch (const exception&). Since out_of_range is-a exception, a base-first ordering would swallow everything in the first block and the specific handler would be unreachable (the compiler warns). 
try { risky(); }
catch (const out_of_range& e) { /* specific */ }
catch (const exception& e)   { /* catch-all fallback */ }
Problem 4 Put it together Apply
Sketch how a Controller reading a bad menu choice from a file should use exceptions so that the View shows a friendly message but the Model stays consistent.
Answer The Model methods throw on invalid operations (and do nothing else, so state is unchanged on failure — the strong guarantee). The Controller wraps each Model call in try/catch; on catch it asks the View to print the error and re-prompts. The View never touches exceptions itself. 
try {
    model.applyChoice(parseFromFile(line));  // Model validates, may throw
} catch (const exception& e) {
    view.showError(e.what());                 // View only displays
}
Because the Model throws before mutating anything, a rejected operation leaves the data exactly as it was — no half-applied state.
