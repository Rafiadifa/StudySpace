Q14Easy2 ptsadding virtual
Fill in both blanks so draw() uses dynamic dispatch and memory is cleaned up safely.
class Shape { public: _____ void draw() { cout << "Shape"; } _____ ~Shape() = default; };
Self-grade:
Answer Both blanks: virtual

Why both? virtual on draw() enables dynamic dispatch. virtual on ~Shape() ensures that when you do delete shapePtr through a base pointer, C++ calls the correct derived destructor first — without it, the derived destructor is silently skipped (memory leak / UB).
Q15Easy3 ptsabstract class + override
Make Shape abstract and complete Circle. Fill in the 3 blanks.
class Shape { public: virtual double area() ____; // pure virtual virtual ~Shape() = default; }; class Circle : ____ Shape { double r; public: Circle(double r) : r(r) {} double area() ____ { return 3.14 * r * r; } };
Self-grade:
Answer Blank 1: = 0  ·  Blank 2: public  ·  Blank 3: override

= 0 makes area() pure virtual → Shape becomes abstract (cannot be instantiated). public means Circle "is-a" Shape — the public interface is preserved. override asks the compiler to verify the signature matches exactly — catches typos like missing const.

Answer

Q18Medium3 ptsmissing virtual destructor
This code has a memory leak / undefined behavior. Find it, explain it, write the fix.
class Base { public: ~Base() { cout << "~Base"; } // <-- problem here? virtual void run() = 0; }; class Derived : public Base { int* data_ = new int[100]; public: ~Derived() { delete[] data_; } void run() override {} }; Base* p = new Derived(); delete p;
Self-grade:
Answer Bug: ~Base() is not virtual.

Why it's wrong: delete p through a Base* uses static binding to find the destructor. Without virtual, only ~Base() runs. ~Derived() is never called, so data_ (100 ints on the heap) is never freed — memory leak and undefined behavior.

Fix: Add virtual: virtual ~Base() { cout << "~Base"; }
Now delete p calls ~Derived() first (frees data_), then ~Base().
Q19Hard4 ptsobject slicing + missing const/override
This code has two bugs. Find both, explain each, write the corrected code.
class Vehicle { public: virtual void describe() const { cout << "Vehicle"; } virtual ~Vehicle() = default; }; class Car : public Vehicle { public: void describe() { cout << "Car"; } // Bug 1? }; Car c; Vehicle v = c; // Bug 2? v.describe();
Self-grade:
Two Bugs Bug 1: Car::describe() is missing const and override. The base declares describe() const, so Car's version with a different signature does NOT override it — it creates a brand new separate function. Adding override would have caught this at compile time.
Fix: void describe() const override { cout << "Car"; }

Bug 2: Vehicle v = c; is object slicing. Copying a derived object by value into a base variable strips all the Derived-specific data and replaces the vtable with Base's — v is now purely a Vehicle. v.describe() prints Vehicle.
Fix: Use a pointer or reference: Vehicle* v = &c; v->describe(); → prints Car.
Q20Expert5 ptsdiamond problem — full fix
This code causes an ambiguity compile error. Identify the root cause and rewrite the three class declarations to fix it correctly.
class Person { public: string name; Person(string n) : name(n) {} }; class Student : public Person { // Bug public: Student(string n) : Person(n) {} }; class Teacher : public Person { // Bug public: Teacher(string n) : Person(n) {} }; class TA : public Student, public Teacher { public: TA(string n) : Student(n), Teacher(n) {} }; TA ta("Rafi"); cout << ta.name; // ambiguous: which Person?
Self-grade:
Answer — Diamond Problem Root cause: Without virtual inheritance, TA contains two separate Person subobjects — one from Student's path and one from Teacher's path. ta.name is ambiguous: the compiler can't decide which copy to use.

Fix — add virtual to both intermediate classes: 
class Student : virtual public Person {
public:
    Student(string n) : Person(n) {}
};
class Teacher : virtual public Person {
public:
    Teacher(string n) : Person(n) {}
};
// TA must now initialize Person directly (most-derived rule):
class TA : public Student, public Teacher {
public:
    TA(string n) : Person(n), Student(n), Teacher(n) {}
};
With virtual inheritance there is exactly one shared Person subobject. ta.name is now unambiguous. The most-derived class (TA) is always responsible for constructing the virtual base — Student and Teacher's calls to Person(n) are ignored.
