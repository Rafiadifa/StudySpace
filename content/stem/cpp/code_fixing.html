Q1 Medium if-else order · Operators
Grade Calculator
Goal: Print the letter grade for a score. A ≥ 90, B ≥ 80, C ≥ 70, D ≥ 60, F otherwise. How many bugs can you find?
Buggy Code
void printGrade(int score) {
    if (score >= 60)            // check 1
        cout << "D";
    else if (score >= 70)       // check 2
        cout << "C";
    else if (score >= 80)       // check 3
        cout << "B";
    else if (score >= 90)       // check 4
        cout << "A";
    else
        cout << "F";
}
Fixed Code
Explanation
Key Rule
Fixed Code
void printGrade(int score) {
    if (score >= 90)      // ✓ highest first
        cout << "A";
    else if (score >= 80)
        cout << "B";
    else if (score >= 70)
        cout << "C";
    else if (score >= 60)
        cout << "D";
    else
        cout << "F";
}
🐛
Bug — wrong order of conditions. The first condition score >= 60 catches every passing score including 90, 80, 70. Because if-else if stops at the first true condition, a score of 95 hits the >= 60 check and prints "D" immediately — the higher checks are never reached.
✅
Fix — order from most restrictive to least. Always place the tightest condition first (>= 90), then progressively loosen. Once a score fails the 90 threshold, the 80 check runs, and so on.
🔍
Trace with score = 95: Buggy: 95 >= 60 → true → prints "D" ✗. Fixed: 95 >= 90 → true → prints "A" ✓
Rule — if-else-if order matters: Conditions are evaluated top-to-bottom and stop at the first true branch. For overlapping ranges (like grade thresholds), always check from the most specific (highest) to the least specific (lowest). A larger number satisfies both >= 90 and >= 60 — only the order decides which branch runs.
Q2 Medium Loops · Off-by-one · Max finding
Find the Largest Value
Goal: Return the largest value in an array of integers. The function has two bugs that make it silently fail for certain inputs.
Buggy Code
int findMax(int arr[], int size) {
    int maxVal = 0;          // bug 1
    for (int i = 0; i <= size; i++) {  // bug 2
        if (arr[i] > maxVal)
            maxVal = arr[i];
    }
    return maxVal;
}
Fixed Code
Explanation
Key Rule
int findMax(int arr[], int size) {
    int maxVal = arr[0];     // ✓ start from first element
    for (int i = 1; i < size; i++) {   // ✓ i < size (not <=)
        if (arr[i] > maxVal)
            maxVal = arr[i];
    }
    return maxVal;
}
🐛
Bug 1 — maxVal = 0. Initializing to 0 silently fails when the array contains only negative numbers (e.g., {-5, -2, -8}). No element beats 0, so the function returns 0 — which is not even in the array. Fix: initialize to arr[0] (the first actual element).
🐛
Bug 2 — i <= size. Valid indices are 0 to size - 1. When i == size, arr[size] reads one position past the end of the array — this is undefined behavior and can crash or silently corrupt data.
✅
Fix — initialize to arr[0], start loop at i = 1, use i < size. Since we've already "looked at" index 0 as the initial max, we start comparing from index 1. The condition i < size stops safely at the last valid index.
Rule 1 — initialize max/min to the first element, not 0: 0 is a magic number that breaks for all-negative arrays. Use arr[0].
Rule 2 — off-by-one with arrays: An array of size n has valid indices 0 … n-1. The correct loop condition is i < size, never i <= size. Reading arr[size] is undefined behavior — no crash guarantee, but results are meaningless.
Q3 Medium while loop · Infinite loop
Sum Until Negative
Goal: Keep reading integers from the user and accumulate a sum. Stop when the user enters a negative number. Return the sum. There are two bugs.
Buggy Code
int sumUntilNegative() {
    int sum = 0;
    int n = 0;
    while (n >= 0) {   // bug 1
        sum = sum + n;
        cin >> n;       // bug 2: cin is inside, but check below
    }
    sum = sum + n;      // accidentally adds the negative number
    return sum;
}
Fixed Code
Explanation
Key Rule
int sumUntilNegative() {
    int sum = 0;
    int n;
    cin >> n;                    // ✓ read before the loop checks
    while (n >= 0) {
        sum = sum + n;           // ✓ only add non-negative numbers
        cin >> n;
    }
    return sum;                  // ✓ negative number never added
}
🐛
Bug 1 — first iteration always adds 0. n starts as 0, and 0 >= 0 is true, so the loop body runs immediately and adds 0 to sum before any user input is read. This doesn't break the sum (adding 0 is harmless) but it's logically wrong — the loop runs once before the user has entered anything.
🐛
Bug 2 — the negative number is added to sum. After the while condition fails (user entered a negative n), the line sum = sum + n runs, adding the negative sentinel to the total. This corrupts the result.
✅
Fix — "prime the pump" pattern. Read the first n before the loop. The while condition then checks it. If it's negative, the loop body never runs (correct). If positive, the body adds it and reads the next n. The negative sentinel is never added.
Rule — "prime the pump" for input loops: When a while loop needs to check a value that comes from user input, read the first value before the loop. This ensures the condition has a real value to test on the very first iteration, and that the sentinel (stop value) is never processed as data.

Q4 Medium Pass by value vs reference · Swap
Swap Two Numbers
Goal: Swap the values of two variables in the caller. After calling swap(a, b), a and b should be exchanged.
Buggy Code
void swap(int x, int y) {   // bug: passes by value
    int temp = x;
    x = y;
    y = temp;
}
int main() {
    int a = 3, b = 7;
    swap(a, b);
    cout << a << " " << b;    // still prints: 3 7
}
Fixed Code
Explanation
Key Rule
void swap(int& x, int& y) {   // ✓ pass by reference
    int temp = x;
    x = y;
    y = temp;
}
int main() {
    int a = 3, b = 7;
    swap(a, b);
    cout << a << " " << b;    // now prints: 7 3
}
🐛
Bug — parameters are passed by value. When you call swap(a, b), C++ copies the values of a and b into new local variables x and y. The swap logic correctly swaps x and y — but those are local copies. When the function returns, the copies are destroyed. a and b in main are completely unchanged.
✅
Fix — use reference parameters: int& x. The & makes x a reference (alias) to the original variable. x and a now point to the exact same memory location. Any change to x directly changes a.
🔍
Memory picture: By value: a=3 at address 0x100, x=3 (copy) at address 0x200. Swap changes 0x200 only. By reference: x IS 0x100 — same box as a. Swap changes 0x100, so a changes.
Rule — pass by reference (&) when you need to modify the caller's variable: Pass by value = copy. Any changes stay local. Pass by reference (int& x) = alias to original. Changes affect the caller. If you only need to read, pass by value or const&.
Q5 Hard Return type · Integer division · Recursive logic
Average of Three
Goal: Return the average of three integers as a decimal value (e.g., average of 1, 2, 3 should be 2.0). The function silently produces wrong results. Find all bugs.
Buggy Code
int average(int a, int b, int c) {  // bug 1: return type
    return (a + b + c) / 3;          // bug 2: integer division
}
int main() {
    cout << average(1, 2, 4);         // expected 2.33, prints 2
}
Fixed Code
Explanation
Key Rule
double average(int a, int b, int c) {  // ✓ return double
    return (a + b + c) / 3.0;            // ✓ 3.0 forces floating-point division
}
Common mistake — this is still wrong:
double result = (a + b + c) / 3;
The division happens first as integer division, then the result (already truncated) is converted to double. You still get 2.0 instead of 2.33. The fix must happen inside the division itself.
🐛
Bug 1 — return type is int. Declaring int average() tells the compiler this function returns an integer. Even if the division somehow produced 2.33, it would be truncated to 2 before returning.
🐛
Bug 2 — integer division: / 3. Both sides of the division are integers (a+b+c is an int, and 3 is an int literal). C++ performs integer division, which truncates the decimal. 7 / 3 = 2, not 2.33.
✅
Fix — change divisor to 3.0 (a double literal) and return type to double. When at least one operand is a floating-point type, C++ promotes the whole division to floating-point. 7 / 3.0 = 2.333.... Alternatively, cast: (double)(a + b + c) / 3.
Rule — integer division truncates, always: If both operands are int, the result is int (decimal discarded). To get a decimal result, at least one operand must be floating-point: use 3.0, or cast with (double) before the division, not after.
Rule — return type must match what you return: A function declared int f() that returns 2.33 will silently truncate to 2. Always match the return type to the expected precision.
Q6 Hard Recursion · Base case · Stack overflow
Factorial (Recursive)
Goal: Compute n! (factorial) recursively. factorial(5) should return 120. The function has two bugs — one causes wrong results, one causes infinite recursion.
Buggy Code
int factorial(int n) {
    if (n == 1)              // bug 1: base case misses n = 0
        return 1;
    return n * factorial(n);  // bug 2: should be factorial(n - 1)
}
Fixed Code
Explanation
Key Rule
int factorial(int n) {
    if (n <= 1)                      // ✓ handles both n=0 and n=1
        return 1;
    return n * factorial(n - 1);  // ✓ recurse with n-1
}
🐛
Bug 1 — base case only checks n == 1. Mathematically, 0! = 1. If someone calls factorial(0), the base case fails and the function calls factorial(-1), then factorial(-2), forever. The base case should be n <= 1 to handle both 0 and 1.
🐛
Bug 2 — factorial(n) instead of factorial(n - 1). This is infinite recursion. The function calls itself with the same argument n — it never gets closer to the base case. The call stack fills up and crashes (stack overflow). The fix is factorial(n - 1), which reduces n by 1 each time, eventually reaching the base case.
🔍
Trace for factorial(3): factorial(3) → 3 * factorial(2) → 2 * factorial(1) → returns 1 → 2 * 1 = 2 → 3 * 2 = 6 ✓
Rule — every recursive function needs two things:
1. A base case that stops recursion (and handles edge cases like 0).
2. A recursive case that moves toward the base case — the argument must change in a way that eventually reaches the base (e.g., n - 1). Without (1) or (2), you get infinite recursion and a stack overflow crash.
Q7 Hard String · char comparison · Loop logic
Count Vowels in a String
Goal: Return the number of vowels (a, e, i, o, u — case insensitive) in a string. The function misses uppercase vowels and has a logic bug in its condition.
Buggy Code
int countVowels(string s) {
    int count = 0;
    for (int i = 0; i < s.length(); i++) {
        if (s[i] == 'a' | s[i] == 'e' |   // bug 1: | not ||
            s[i] == 'i' | s[i] == 'o' |
            s[i] == 'u')                   // bug 2: no uppercase
            count++;
    }
    return count;
}
Fixed Code
Explanation
Key Rule
int countVowels(string s) {
    int count = 0;
    for (int i = 0; i < s.length(); i++) {
        char c = tolower(s[i]);     // ✓ normalize to lowercase
        if (c == 'a' || c == 'e' ||  // ✓ logical OR (||)
            c == 'i' || c == 'o' ||
            c == 'u')
            count++;
    }
    return count;
}
🐛
Bug 1 — single | (bitwise OR) instead of || (logical OR). | performs a bitwise operation on the integer values of the characters. It will compile without error but produces incorrect boolean logic — it may count non-vowels or miss vowels because it's operating on bit patterns, not true/false values. Always use || to combine boolean conditions.
🐛
Bug 2 — no uppercase handling. The condition only checks lowercase 'a' through 'u'. A string like "Apple" has 'A' and 'e' = 2 vowels, but the function returns 1 (misses 'A'). Fix: convert each character to lowercase with tolower() from <cctype> before comparing.
Rule — | vs ||: | = bitwise OR (operates on individual bits of integers). || = logical OR (combines boolean true/false). For combining conditions in if statements always use || and &&, not | and &. The bitwise versions compile but give logically wrong results with booleans.
Rule — case-insensitive comparisons: Use tolower(c) (from #include <cctype>) to normalize a character to lowercase before comparing. This way you only need to check lowercase variants.

Q8 Hard Class · Encapsulation · const · Setter validation
Temperature Class
Goal: A Temperature class stores Celsius degrees. getTemp() should not modify the object. setTemp() should reject values below absolute zero (−273.15°C). The class has three design bugs.
Buggy Code
class Temperature {
public:                           // bug 1: data should be private
    double celsius;

    double getTemp() {             // bug 2: should be const
        return celsius;
    }

    void setTemp(double t) {
        celsius = t;               // bug 3: no validation
    }
};
Fixed Code
Explanation
Key Rule
class Temperature {
private:                               // ✓ data hidden from outside
    double celsius;

public:
    Temperature(double t = 0.0) : celsius(t) {}

    double getTemp() const {          // ✓ const: won't modify object
        return celsius;
    }

    void setTemp(double t) {
        if (t >= -273.15)              // ✓ validate before setting
            celsius = t;
        else
            cout << "Invalid: below absolute zero\n";
    }
};
🐛
Bug 1 — celsius is public. Any code can write obj.celsius = -9999, completely bypassing the validation in setTemp(). The whole point of a setter is to control access — but it's useless if the data is public. Move data to private.
🐛
Bug 2 — getTemp() is not const. A getter should never modify the object — marking it const enforces this. More importantly, if you have a const Temperature object, you cannot call non-const methods on it at all, making the object essentially read-only but inaccessible. Always mark read-only methods const.
🐛
Bug 3 — no validation in setTemp(). The setter assigns any value, including physically impossible temperatures below −273.15°C (absolute zero). A setter's purpose is to guard the data. Add an if check before assigning.
Rule — Encapsulation checklist:
• Data members → private
• Getters → const (they only read)
• Setters → validate before assigning
• This way the class controls its own state and prevents invalid data.
Q9 Hard Constructor · Initializer list · Static member · Definition
Student Counter Class
Goal: Each Student object gets a unique ID (starting from 1). A static counter tracks how many students exist. The class has three bugs — one prevents compilation, one gives wrong IDs, one corrupts the count on destruction.
Buggy Code
class Student {
    int id;
    static int count;
public:
    Student() {
        count++;
        id = count;              // seems right...
    }
    ~Student() { count++; }   // bug 1: should decrement
    static int getCount() { return count; }
};
// bug 2: static member never defined outside class
// bug 3: if two Students are created in same expression,
//        id assignment order vs count increment is fragile.
//        Better practice: id = ++count (pre-increment)
Fixed Code
Explanation
Key Rule
class Student {
    int id;
    static int count;
public:
    Student() {
        id = ++count;            // ✓ pre-increment: count becomes 1, id = 1
    }
    ~Student() { count--; }   // ✓ decrement on destroy
    static int getCount() { return count; }
    int getId() const { return id; }
};

int Student::count = 0;       // ✓ must define static outside class
🐛
Bug 1 — destructor increments instead of decrements (count++). When a Student object is destroyed, the count should decrease (one fewer student alive). Using count++ makes the count grow on destruction, completely defeating the purpose of tracking live students.
🐛
Bug 2 — static member count declared but never defined. In C++, static data members must be declared inside the class AND defined (given storage) outside the class with int Student::count = 0;. Without the definition, the linker throws "undefined reference" and the code won't compile.
🐛
Bug 3 — count++; id = count; uses post-increment with separate assignment. This works here, but id = ++count is cleaner and safer: it increments count first, then assigns that new value to id — all in one expression, guaranteeing id 1 for the first student, 2 for the second, etc.
Rule — static members need an out-of-class definition:
Inside class: static int count; — this is only a declaration.
Outside class: int Student::count = 0; — this is the definition (allocates actual memory).
Missing the definition = linker error every time.
Q10 Expert Inheritance · virtual destructor · Base constructor · Override
Animal & Dog Inheritance
Goal: Dog inherits from Animal. Deleting via a base pointer should call both destructors. speak() on a base pointer should call Dog's version. The class has four bugs.
Buggy Code
class Animal {
public:
    string name;                        // bug 1: should be private/protected
    Animal(string n) { name = n; }
    ~Animal() { cout << "~Animal "; }  // bug 2: not virtual
    void speak() { cout << "..."; }    // bug 3: not virtual
};

class Dog : public Animal {
public:
    Dog(string n) { name = n; }         // bug 4: doesn't call base ctor
    ~Dog() { cout << "~Dog "; }
    void speak() { cout << "Woof!"; }
};

int main() {
    Animal* a = new Dog("Rex");
    a->speak();   // prints "..." instead of "Woof!"
    delete a;    // only calls ~Animal, leaks ~Dog
}
Fixed Code
Explanation
Key Rule
class Animal {
protected:                                    // ✓ accessible to Dog, not public
    string name;
public:
    Animal(string n) : name(n) {}             // ✓ initializer list
    virtual ~Animal() { cout << "~Animal "; } // ✓ virtual destructor
    virtual void speak() { cout << "..."; }  // ✓ virtual for polymorphism
};

class Dog : public Animal {
public:
    Dog(string n) : Animal(n) {}            // ✓ delegate to base ctor
    ~Dog() { cout << "~Dog "; }
    void speak() override { cout << "Woof!"; }
};

int main() {
    Animal* a = new Dog("Rex");
    a->speak();   // ✓ now prints "Woof!"
    delete a;    // ✓ calls ~Dog then ~Animal
}
🐛
Bug 1 — name is public. Any outside code can write animal.name = "hacked" with no control. Use protected: it hides name from outside code while still letting Dog and other derived classes access it directly.
🐛
Bug 2 — destructor is not virtual. When you delete an object through a base class pointer (Animal* a = new Dog(...)), C++ only calls ~Animal — it has no idea a Dog is underneath. The Dog destructor never runs, causing resource leaks. Adding virtual to ~Animal tells C++ to look up the actual destructor at runtime.
🐛
Bug 3 — speak() is not virtual. Without virtual, calling a->speak() on an Animal* always calls Animal::speak() — the pointer type decides, not the actual object. Adding virtual enables runtime dispatch: C++ checks the actual object (Dog) and calls Dog::speak().
🐛
Bug 4 — Dog constructor doesn't call the base constructor. Animal has no default (no-argument) constructor. Dog(string n) { name = n; } tries to default-construct Animal first (impossible), then assign name directly (which works only because name is public/protected). The correct pattern is to use the initializer list: Dog(string n) : Animal(n) {}, which explicitly calls Animal's constructor with the name.
Rule — always use virtual destructor in a base class: If a class is meant to be inherited and objects will be deleted through base pointers, the base destructor must be virtual. Without it, derived destructors are silently skipped — a hard-to-detect resource leak.
Rule — virtual enables runtime polymorphism: Without virtual, function calls are resolved at compile time by the pointer/reference type. With virtual, they are resolved at runtime by the actual object type. Add override in the derived class — the compiler will verify the signature matches.
Rule — call base constructor in derived initializer list: If the base class has no default constructor, derived classes must call the base constructor explicitly in their initializer list: Derived(args) : Base(args) {}.
