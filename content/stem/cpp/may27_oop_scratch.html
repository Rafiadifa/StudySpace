Recall 1 Anatomy of a class
⌁ Do this from memory
Write a BankAccount class with a private owner (string) and balance (double). Include: a constructor, a deposit, a withdraw that refuses overdrafts, and a const getter for the balance.
  • Which members are private vs public, and why?
  • Why is the getter marked const?
Model 
class BankAccount {
private:
    string owner;
    double balance;
public:
    BankAccount(string o, double b) : owner(o), balance(b) {}

    void deposit(double amt) { if (amt > 0) balance += amt; }

    bool withdraw(double amt) {
        if (amt > 0 && amt <= balance) { balance -= amt; return true; }
        return false;            // refuse overdraft
    }

    double getBalance() const { return balance; }
};
Encapsulation: data is private so it can only change through validated methods. The getter is const because reading the balance must never modify the object — and only const methods are callable on a const BankAccount.
Recall 2 Constructors & the initializer list
⌁ Do this from memory
Without looking: what's the difference between initializing members in the initializer list versus assigning them in the constructor body? Write a small class that must use the initializer list, and say why.
Model The initializer list constructs each member once, directly. Body assignment first default-constructs the member, then overwrites it. Three cases require the list: const members, reference members, and members with no default constructor. 
class Sensor {
    const int id;     // const → must be in the list
    int& reading;     // reference → must be in the list
public:
    Sensor(int i, int& r) : id(i), reading(r) {}
};
A const or reference member can never be assigned after construction, so it has to be set in the initializer list. Members are also initialized in declaration order, not list order — a classic gotcha.
Recall 3 The Big Picture — 4 pillars
⌁ Explain aloud (Feynman)
Name the four pillars of OOP and give a one-line C++ mechanism for each. Pretend you're teaching a friend who knows C but not C++.
Model Encapsulation — bundle data + behaviour, hide internals with private. Abstraction — expose a clean interface; hide the "how" (public methods, pure virtuals). Inheritance — class Derived : public Base reuses and extends. Polymorphism — one interface, many behaviours via virtual functions dispatched at runtime.
If you can give a concrete C++ keyword for each pillar (private, interface, : public, virtual), you've got the conceptual scaffold the final tests.
